- Improve indexability.

- Support structured metadata.

# URL Structure

All public URLs shall be:

- Human readable
- Short
- Descriptive
- Consistent
- Lowercase
- Hyphen separated

Examples:

```text
/

about

services

gallery

previous-events

dream-planner

booking-consultation

contact
```

URLs should remain stable after publication.

---

# URL Best Practices

URLs shall:

- Avoid unnecessary parameters.
- Avoid uppercase characters.
- Avoid special symbols.
- Avoid excessive nesting.
- Avoid session identifiers.

Example:

Good

```text
/services/weddings
```

Poor

```text
/services?id=53&type=wedding
```

---

# Website Hierarchy

The public website shall follow a shallow hierarchy.

```text
Home

├── About

├── Services

├── Gallery

├── Previous Events

├── Dream Planner

├── Booking Consultation

└── Contact
```

Important pages should be accessible within a few clicks from the homepage.

---

# Crawlability

Search engine crawlers shall be able to discover all important public pages.

The platform shall avoid:

- Broken internal links
- Orphan pages
- Infinite crawl paths
- Blocked public resources

Navigation should support efficient crawling.

---

# Robots.txt

A robots.txt file shall be provided.

It should:

- Allow indexing of public content.
- Block administrative routes.
- Block development resources.
- Reference the XML sitemap.

Example:

```text
User-agent: *

Disallow: /admin/

Disallow: /dashboard/

Sitemap: https://example.com/sitemap.xml
```

---

# XML Sitemap

The platform shall automatically generate an XML sitemap.

The sitemap should include:

- Homepage
- About
- Services
- Gallery
- Previous Events
- Dream Planner
- Booking Consultation
- Contact

Future content should automatically appear within the sitemap.

---

# Sitemap Best Practices

The sitemap should:

- Exclude duplicate URLs.
- Exclude administrative pages.
- Exclude private resources.
- Include canonical URLs only.
- Update automatically after content changes.

---

# Indexability

Public pages intended for search visibility shall remain indexable.

Pages not intended for indexing include:

- Admin Dashboard
- Login Pages
- Error Pages
- Development Routes
- Temporary Preview Pages

Indexing behavior should be explicitly controlled.

---

# Meta Robots Directives

Supported directives include:

- index
- noindex
- follow
- nofollow

Examples:

Public Page

```html
<meta name="robots" content="index, follow">
```

Private Page

```html
<meta name="robots" content="noindex, nofollow">
```

---

# Canonical URLs

Every indexable page shall define a canonical URL.

Example:

```html
<link rel="canonical"
href="https://matchstickevents.com/services">
```

Canonical tags reduce duplicate content issues.

---

# Duplicate Content Prevention

The platform shall minimize duplicate content through:

- Canonical URLs
- Single preferred URL structure
- Redirects
- Consistent internal linking

Every piece of content should have one authoritative URL.

---

# Redirect Strategy

Redirects shall preserve SEO value.

Preferred redirect types:

- 301 (Permanent)
- 308 (Permanent)

Temporary redirects should only be used when appropriate.

Redirect chains should be avoided.

---

# HTTPS

Every public page shall use HTTPS.

HTTP requests shall redirect permanently to HTTPS.

Benefits include:

- Security
- User Trust
- Improved Search Ranking
- Modern Browser Compatibility

Mixed content shall not be permitted.

---

# WWW Strategy

A single preferred domain shall be selected.

Examples:

```text
https://matchstickevents.com
```

or

```text
https://www.matchstickevents.com
```

All alternatives shall redirect permanently to the preferred domain.

---

# Breadcrumbs

Where applicable, pages should support breadcrumb navigation.

Example:

```text
Home

↓

Services

↓

Weddings
```

Breadcrumbs improve both usability and search engine understanding.

---

# Internal Linking

Internal links shall:

- Connect related content.
- Support logical navigation.
- Improve crawl efficiency.
- Distribute page authority.

Every important page should receive internal links.

---

# Pagination

If pagination is introduced in future:

- Maintain crawlable links.
- Preserve logical navigation.
- Avoid duplicate content.
- Support canonicalization where appropriate.

---

# Error Pages

404 pages shall:

- Return HTTP 404 status.
- Provide navigation back to the website.
- Preserve brand consistency.
- Avoid indexing.

Custom error pages should improve user experience.

---

# Broken Link Prevention

The platform should periodically verify:

- Internal Links
- External Links
- Images
- Documents
- Media Assets

Broken links should be corrected promptly.

---

# Crawl Budget Optimization

Although the website is relatively small, crawl efficiency should still be maintained.

Avoid:

- Duplicate URLs
- Infinite filters
- Excessive redirects
- Thin pages
- Unnecessary parameters

Efficient crawling supports faster indexing.

---

# Future Scalability

The Technical SEO architecture shall support:

- Blog Pages
- City Landing Pages
- Service Landing Pages
- Portfolio Expansion
- Multilingual Support
- Additional Media Content

The URL architecture should scale without major restructuring.

---

# Non-Functional Requirements

The Technical SEO architecture shall be:

- Crawlable.
- Indexable.
- Maintainable.
- Scalable.
- Secure.
- High Performance.
- Search Engine Friendly.
- Production Ready.

---

# Developer Notes

Developers should:

- Build every public page using semantic HTML, stable URL structures, proper HTTP status codes, and crawl-friendly navigation to maximize search engine accessibility.
- Generate robots.txt, XML sitemaps, canonical tags, and metadata automatically wherever possible to reduce manual maintenance and ensure long-term consistency.
- Prevent duplicate content by enforcing a single preferred URL for every resource and minimizing unnecessary redirects, query parameters, and alternate page versions.
- Continuously monitor crawl errors, broken links, indexing status, and sitemap health as part of routine website maintenance.
- Design the technical SEO foundation to accommodate future additions—including blogs, city-specific pages, multilingual support, portfolios, and expanded service offerings—without requiring significant architectural changes.
